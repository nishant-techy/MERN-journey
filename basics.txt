*Align-items property*


The align-content property modifies the behavior of the flex-wrap property. It is similar to align-items, but instead of aligning flex items, it aligns flex lines. Property values:
center lines stretch to take up the remaining space.

  display: flex;
  flex-wrap: wrap;
  align-content: center;
}

flex-start lines are packed toward the start of the flex container.

.wrapper {
  display: flex;
  flex-wrap: wrap;
  align-content: flex-start;
}

flex-end lines are packed toward the end of the flex container.

.wrapper {
  display: flex;
  flex-wrap: wrap;
  align-content: flex-end;
}

space-between lines are evenly distributed in the flex container.

.wrapper {
  display: flex;
  flex-wrap: wrap;
  align-content: space-between;
}

space-around lines are evenly distributed in the flex container, with half-size spaces on either end.

.wrapper {
  display: flex;
  flex-wrap: wrap;
  align-content: space-around;

i mean if we want to add the function to the every button who have the class "done", then if we create another button later by any other method without changing the code (like dynamic buttons) or something like that, then those function will not apply on these new buttons,

aur isme kya hota h ki jo function/feature hme uss naye element pe lgana tha to uspe wo apply nhi hota to instead of doing on that, we'll apply the same for our parent element,

------------------Async vs await------------------
1. async: used for telling the JS that this function will work with asynchronous operations and will always return a Promise.

When you put async before a function:
async function getFact() {
}

Example:

async function hello() {
    return "Hello";
}

it will prints "Hello"




2. Await: pause this async function until the promise finishes

Example:
let response = await fetch(url);
JavaScript waits for the API request to complete before moving to the next line.


Without await:
let response = fetch(url);
console.log(response);

You'll see a Promise object instead of the actual result.


*await* can only be used::
Inside an async function, or
At top level in environments that support top-level await.

So::
async function getFact() {
    let response = await fetch(url);
}

is valid.

---------------- Import Module ----------------

Before the ES6 update, Node.js mainly used the `require()` method to import modules. When we use `require()`, the whole module file is loaded and executed. Even if we only need a few functions from that file, the complete module is loaded.

To improve module handling, ES6 introduced the `import` and `export` syntax.

One advantage of `import` is that we can import only the things we need from a module. For example, if a file contains thousands of lines of code and many functions, but we only need two or three functions, we can import only those specific functions.

```js
import { add, subtract } from './math.js';
```

In CommonJS (`require`), we can also access specific functions, but ES6 modules allow tools like Webpack and Vite to perform tree shaking, which can remove unused code and make the final bundle smaller and more efficient.

Another difference is that `require()` loads modules synchronously, while `import` supports asynchronous loading through dynamic imports.

To use ES6 modules in Node.js, we need to add the following in the `package.json` file:

```json
{
  "type": "module"
}
```


If our project does not have a `package.json` file, we can create one by running:

```bash
npm init -y
```

After creating the `package.json` file, we can add `"type": "module"` and then use `import` and `export` syntax in our project properly.

----- cleaner definition -----

1). Path Parameters are dynamic values passed inside the URL path. They allow us to create routes that can handle different values using a single route definition.

Example:
/ig/viratkohli
/ig/cristiano
/ig/selena_gomez
Here, the username part changes every time, so we use a path parameter.


2). Route vs Path Parameters

 Routes: Routes is the something that i just explain above
Actually:
Route → /ig/:username
Path Parameter → :username
So the route contains a path parameter.



Why are the curly braces used?
You asked:
why would we place those brackets around the variable?
This is called object destructuring.

Without destructuring:
let username = req.params.username;

With destructuring:
let { username } = req.params;

Both do exactly the same thing.










In the above route:

/ig/:username is the route.
:username is the path parameter.

req.params contains all path parameters.

let { username } = req.params uses object destructuring to extract the username value.

For example:
/ig/viratkohli
will result in:

req.params = {
    username: "viratkohli"
}

and the value "viratkohli" will be stored inside the username variable.
